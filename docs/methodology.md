# 調査方法論 — なぜこの順番で、何を見るのか

> この文書は Skill の背後にある考え方を説明する。Skill が「手順書」なら、これは「なぜその手順なのか」。

## 大原則：改造の前に理解

レガシーシステムの事故の多くは、「分かったつもり」で手を入れることから始まる。
このワークフローの成果物は修正コードではなく、**正確な As-Is の地図**である。

地図がないまま見積もると何が起きるか：

- テーブル数や画面数だけで工数を弾く → 方言依存や自作フレームワークの存在で爆発する
- 「動いているコード＝正しい仕様」と誤読する → 暗黙の業務ルール（バグ込みの挙動に依存した運用）を壊す

## なぜ「棚卸し → 構造 → データ層 → 報告」の順か

1. **棚卸しが最初**なのは、フレームワークとバージョンが「その後の全調査の探し方」を決めるから。Struts と Spring ではエントリポイントの探し方が違う。ここを誤ると以降の調査が全部ずれる。
2. **構造が二番目**なのは、エントリポイントと依存の地図がないと、データ層調査で見つけた問題の「影響範囲」が語れないから。
3. **データ層が三番目**なのは、マイグレーション工数の分散が最も大きい（=見積もりを最も外しやすい）領域だから。独立した評価軸（方言依存の濃度）で測る。
4. **報告書が最後**なのは当然だが、重要なのは**検収ゲートを挟む**こと。中間生成物の矛盾や根拠なし数値を、報告書に流入させない。

## AI に調査させるときの設計判断

このリポジトリの Skill 設計で意識していること：

### 1. 中間生成物の形式が成否を決める

マルチステップの調査で最も壊れやすいのは、ステップ間の受け渡しである。

- 生ログをそのまま次工程に渡す → コンテキストが溢れ、後段の品質が落ちる
- 要約しすぎる → 根拠が失われ、報告書が「それっぽい作文」になる

そこで全フェーズに**共通の規律**を課している：結論には根拠ファイルを併記、数値はコマンド実行結果のみ、不明点は「未確認」として確証度付きで別枠管理。

### 2. 「未確認」を一級市民にする

AI は空欄を埋めたがる。実務の調査報告で最も危険なのは、**もっともらしく埋められた空欄**である。
全 Skill の出力テンプレートに「未確認事項」セクションを必須で置いているのはこのため。未確認の管理こそが、AI 調査を実務品質にする鍵。

### 3. 全数調査と代表調査を使い分ける

- **全数**: エントリポイント、方言依存 SQL — 機械的に数えられて、漏れが事故になるもの
- **代表**: 処理フローの追跡 — 全部追うのはコストに見合わない。「典型的な書き方」を掴めばよい

この使い分けを Skill に明記しておかないと、AI は際限なく全部を読もうとするか、逆にサンプリングで済ませて漏らすかのどちらかに振れる。

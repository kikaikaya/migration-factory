# 調査報告書の生成

中間生成物を「読んだ人が意思決定できる文書」に変換する。**調査メモの寄せ集めは報告書ではない。**

## 前提

`survey-output/01-inventory.md`〜`03-database.md` をすべて読み込む。欠けているフェーズがあれば、報告書に「未実施」と明記する（勝手に補完しない）。

## 報告書の構成

`survey-output/00-report.md` に以下の構成で生成する：

```markdown
# 〇〇システム 現状調査報告書

## 1. エグゼクティブサマリー（1ページ以内・専門用語禁止）
   - このシステムは何で、どんな状態か（3行）
   - 最重要リスク Top3（ビジネスインパクトで書く。技術用語で書かない）
   - 推奨アクション（次の一手）

## 2. システム概要（規模・構成・時代背景）

## 3. 技術スタックと EOL リスク（01 から要約 + 表）

## 4. 構造評価（02 から要約。公称 vs 実態のギャップを中心に）

## 5. データ層と移行評価（03 から要約 + 難易度スコア）

## 6. リスク一覧（全章横断・優先度順の表）
   | # | リスク | 影響 | 確証度 | 推奨対応 |

## 7. 改善・移行に向けた提言（As-Is 報告の最終章としてのみ）

## 付録: 未確認事項一覧（人間による追加確認が必要な項目）
```

## 書き方のルール

1. **二層構造を守る**: 1章は役員が読む。2章以降は開発者が読む。1章に技術用語を持ち込まない（「Struts1」ではなく「サポートが終了した10年以上前の技術」と書く）。
2. **数字には根拠を**: 中間生成物にない数字を書かない。すべての数値は 01〜03 から引用できること。
3. **確証度を引き継ぐ**: 中間生成物で「推測」だったものを、報告書で事実に昇格させない。
4. **リスクはビジネス語で**: 「循環依存がある」ではなく「一箇所の修正が予期せぬ別機能の障害を引き起こしやすい構造」と書く。
5. **提言は最終章のみ**: 調査の本分は正確な As-Is。To-Be を語るのは7章だけ。

# データ層調査・DB 移行評価

データ層はマイグレーションで**最も工数見積もりを外しやすい領域**。「テーブル数」ではなく「方言依存の濃度」で難易度を測る。

## 手順

### 1. データアクセス方式の特定

コードベースから実際のアクセス方式を特定する（複数併用が普通）：

- O/R マッパー（MyBatis/iBATIS, Hibernate, 自作 DAO 等）と、その設定ファイルの場所
- 生 SQL の所在（XML / アノテーション / 文字列連結 / プロパティファイル）
- ストアドプロシージャ・ファンクション・トリガーの有無と数

### 2. SQL 資産の棚卸し

SQL を機械的に収集し、件数を計測する：

- SELECT / INSERT / UPDATE / DELETE / MERGE の件数
- **動的 SQL**（文字列連結で組み立てている箇所）の件数 — 静的解析できない分、移行リスクが高い

### 3. 方言依存チェック（Oracle → PostgreSQL の主要チェックポイント）

収集した SQL に対して、以下を grep ベースで全数チェックし、件数と代表例を記録する：

| チェック項目 | Oracle | PostgreSQL 側の対応 |
|--------------|--------|---------------------|
| 階層クエリ | `CONNECT BY` | 再帰 CTE（`WITH RECURSIVE`）への書き換え。**性能特性が変わるため要検証** |
| 外部結合旧記法 | `(+)` | ANSI JOIN への書き換え |
| 関数 | `NVL` / `DECODE` / `SYSDATE` / `TO_DATE` 書式 | `COALESCE` / `CASE` / `CURRENT_TIMESTAMP` 等 |
| 型 | `NUMBER` / `VARCHAR2` / `DATE`（時刻含む） | `NUMERIC` / `VARCHAR` / `TIMESTAMP` — **DATE 型の暗黙の時刻が最頻出の罠** |
| シーケンス | `seq.NEXTVAL` | `nextval('seq')` 記法差 |
| 空文字と NULL | `'' = NULL` 扱い | PostgreSQL は別物。**アプリロジックに波及する** |
| ROWNUM | `ROWNUM` | `LIMIT` / ウィンドウ関数 |
| MERGE | `MERGE INTO` | `INSERT ... ON CONFLICT`（意味差あり） |
| ストアド | PL/SQL | PL/pgSQL 書き換え or アプリ側へ吸い上げ。**件数×複雑度で別見積もり** |
| ヒント句 | `/*+ ... */` | 無効。性能チューニングのやり直し前提 |

### 4. 難易度スコアリング

以下の観点で 5 段階評価し、根拠（件数）を付ける：

- 方言依存 SQL の総量と密度（SQL 100 本あたりの要書き換え件数）
- ストアドの量と複雑度
- 動的 SQL の比率
- DATE 型・空文字 NULL 等「アプリロジックに波及する」項目の有無 ← **ここが高いと SQL 書き換えでは済まない**

### 5. 出力

`survey-output/03-database.md` に以下の構成で出力：

```markdown
# データ層調査・移行評価

## データアクセス方式（方式別の所在マップ）
## SQL 資産サマリー（件数表）
## 方言依存チェック結果（チェックポイント別件数 + 代表例）
## 移行難易度評価（観点別スコア + 総合所見）
## アプリロジック波及リスク（SQL 書き換えで済まない箇所）
## 未確認事項
```

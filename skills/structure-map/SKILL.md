# 構造・依存関係の可視化

「このシステムはどこから始まり、どう流れ、どこで終わるか」を地図にする。

## 前提

`survey-output/01-inventory.md` があれば**先に読む**。フレームワークが分からないままエントリポイントを探すと誤る（Struts なら `struts-config.xml`、Spring MVC ならアノテーション、Servlet なら `web.xml`——探し方が全部違う）。

## 手順

### 1. エントリポイントの全数調査

種類別に**漏れなく**数え上げる（grep ベースで機械的に）：

- **オンライン**: 画面/API のエントリ（フレームワーク設定 or アノテーションから抽出）
- **バッチ**: main メソッド保持クラス、ジョブ定義から呼ばれるシェル/クラス
- **非同期**: MQ リスナー、スケジューラ（Quartz 等）、トリガー

各エントリポイントは「ID / 種類 / 定義場所 / 推定される業務機能名」の表にする。

### 2. レイヤー構造の判定

パッケージ構成と依存方向から、実際のアーキテクチャを判定する：

- 公称アーキテクチャ（パッケージ名が示す設計意図: controller/service/dao 等）
- **実態**（レイヤー飛ばし・逆方向依存・神クラスの有無）

公称と実態のギャップこそが移行リスク。ギャップ箇所は具体的なクラス名で記録する。

### 3. 依存の可視化

- モジュール間依存を Mermaid のグラフで描く（循環依存は赤で強調）
- 「最も依存されているクラス」上位10（= 触ると影響が大きい箇所）
- 共通ライブラリ/ユーティリティの自作率（自作フレームワークの存在は移行難易度を大きく上げる）

### 4. 代表フローの追跡

エントリポイントから代表的なもの（オンライン1本・バッチ1本以上）を選び、入口→出口まで処理を追跡して シーケンス図（Mermaid）にする。**全部は追わない**——代表フローで「このシステムの典型的な書き方」を掴むのが目的。

### 5. 出力

`survey-output/02-structure.md` に以下の構成で出力：

```markdown
# 構造・依存関係調査

## エントリポイント一覧（種類別の表 + 総数）
## アーキテクチャ（公称 vs 実態、ギャップ箇所）
## 依存グラフ（Mermaid + 高依存クラス Top10）
## 代表処理フロー（シーケンス図）
## 構造上の懸念（循環依存・神クラス・自作フレームワーク等）
## 未確認事項
```

# 技術スタック棚卸し

対象リポジトリの「何で・いつの時代に・どう作られているか」を確定させる。**ここの誤りは後続調査すべてに波及する**ため、根拠ファイルを必ず記録する。

## 手順

### 1. 一次情報の収集（推測禁止ゾーン）

ビルド定義・設定ファイルから事実だけを拾う：

- **ビルド**: `pom.xml` / `build.gradle` / `build.xml`(Ant) / `Makefile` — 多段モジュールの場合は親子関係も記録
- **言語バージョン**: コンパイラ設定（`maven.compiler.source`、`sourceCompatibility` 等）
- **フレームワーク**: 依存定義から主要フレームワークと**正確なバージョン**を抽出（Spring / Struts / Seasar2 / EJB など。レガシー案件では Struts1 系・Seasar2 など EOL 済みが頻出）
- **アプリサーバ**: `web.xml` のスキーマバージョン、`weblogic.xml` / `ibm-web-bnd.xml` 等のベンダ固有ファイルの有無
- **DB**: JDBC ドライバ依存、接続設定（`*.properties` / JNDI 定義）から DB 製品とバージョンを特定
- **バッチ/ジョブ**: シェルスクリプト、JP1/JCL/cron の定義ファイルの有無と数

### 2. 規模の計測

コマンドで計測する（目視カウント禁止）：

```bash
# 例: 言語別ファイル数・行数
find . -name "*.java" | wc -l
```

- 言語別ファイル数・総行数
- モジュール数
- 最終コミットの分布（直近1年でどこが触られているか = 生きている範囲の推定）

### 3. EOL チェック

特定した主要コンポーネントごとに、サポート状況を分類する：

| 分類 | 意味 |
|------|------|
| 🔴 EOL 済み | ベンダサポート終了。セキュリティパッチなし |
| 🟡 EOL 接近 | 2年以内にサポート終了予定 |
| 🟢 サポート中 | 現行サポート対象 |

EOL 情報が学習データより新しい可能性があるため、確証がない場合は「要確認」と明記する。

### 4. 出力

`survey-output/01-inventory.md` に以下の構成で出力：

```markdown
# 技術スタック棚卸し

## サマリー（表: コンポーネント / バージョン / EOL状況 / 根拠ファイル）
## 規模（言語別 LOC・モジュール構成）
## 時代背景の推定（このスタックが組まれた年代と当時の典型構成）
## 未確認事項（確証度低の項目と、確認方法の提案）
```

**「根拠ファイル」列は必須。** 根拠を示せない項目はサマリーに入れず、未確認事項に回す。

# レガシー現状調査ワークフロー

調査対象リポジトリのルートで実行する。**コードを一切変更しない。読むだけ。**

## 原則

1. **先に全体、後に詳細。** いきなりコードを読まない。まず棚卸し、次に構造、最後に中身。
2. **改造の前に理解。** このワークフローの仕事は「直すこと」ではなく「正確に知ること」。改善提案は報告書の最終章でのみ行う。
3. **すべての中間生成物を Markdown で残す。** 後続フェーズと人間の両方が読めるようにする。
4. **わからないことは「わからない」と書く。** 推測で埋めた調査報告書は実務では事故のもと。確証度を必ず明記する。

## 実行手順

### Phase 0: 準備

1. `survey-output/` ディレクトリを作成する（既存なら日付サフィックスを付けて退避）
2. 対象リポジトリの規模を概算する（ファイル数・言語別行数）。規模が大きい場合（目安: 50万行超）は、ユーザーに調査範囲（特定モジュールに絞るか）を確認する

### Phase 1: 技術スタック棚卸し

`tech-inventory` Skill を実行し、結果を `survey-output/01-inventory.md` に保存する。

### Phase 2: 構造・依存の可視化

`structure-map` Skill を実行する。**Phase 1 の結果（`01-inventory.md`）を先に読み込むこと**——フレームワークが分かっていないと、エントリポイントの特定方法を間違える。結果は `survey-output/02-structure.md` に保存する。

### Phase 3: データ層調査・移行評価

`db-migration-assess` Skill を実行する。Phase 1〜2 の結果を読み込んだ上で行う。結果は `survey-output/03-database.md` に保存する。

DB を使っていないシステムの場合、このフェーズはスキップし、その旨を `03-database.md` に一行記録する。

### Phase 4: 検収ゲート

報告書を書く前に、ここまでの中間生成物を自己点検する：

- [ ] `01〜03` の各ファイルに「未確認」「推測」の項目が確証度付きで明記されているか
- [ ] エントリポイント数・テーブル数など、数値はコマンド実行結果に基づいているか（目視カウント禁止）
- [ ] 矛盾はないか（例: inventory には Oracle とあるのに database 調査が PostgreSQL 前提になっている等）

問題があれば該当フェーズに戻って修正してから先に進む。

### Phase 5: 報告書生成

`survey-report` Skill を実行し、`survey-output/00-report.md` に最終報告書を生成する。

## 完了時

ユーザーに以下を提示する：

- 報告書の場所と、エグゼクティブサマリー（3行）
- 調査中に発見した「最も危険な箇所」トップ3
- 確証度が低く、人間による追加確認が必要な項目一覧
